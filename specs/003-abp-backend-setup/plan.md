# Implementation Plan: ABP Backend Foundation Setup

**Branch**: `003-abp-backend-setup` | **Date**: 2026-03-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-abp-backend-setup/spec.md`

## Summary

Establish a fully running ABP Zero backend connected to PostgreSQL with Swagger UI accessible.
The primary task is migrating the existing SQL Server configuration to PostgreSQL (replacing the
EF Core provider, updating connection strings, deleting SQL Server migrations, and regenerating
a clean initial migration), then verifying the complete stack builds and runs successfully.

## Technical Context

**Language/Version**: C# / .NET 9.0
**Primary Dependencies**: ABP Zero 10.x, Npgsql.EntityFrameworkCore.PostgreSQL 9.0.4, EF Core 9.0.5
**Storage**: PostgreSQL 15+ via Npgsql
**Testing**: Manual — `dotnet build`, Swagger UI, psql / pgAdmin
**Target Platform**: Local development machine
**Project Type**: Web API (ABP Zero modular monolith — 5 layers)
**Performance Goals**: N/A for setup phase
**Constraints**: No production data exists — safe to delete and regenerate all migrations.
Must preserve ABP Zero seed infrastructure (roles, settings, default language).
**Scale/Scope**: Single-tenant (host) for initial setup

## Ordered Implementation Steps

### Step 1 — Validate Folder Structure Compliance

**Description**: Confirm the existing solution structure matches the ABP layering rules defined
in `docs/BACKEND_STRUCTURE.md` before making any code changes.

**Actions**:
- Open the solution and verify the five projects exist:
  `backend.Core`, `backend.Application`, `backend.EntityFrameworkCore`,
  `backend.Web.Core`, `backend.Web.Host`
- Confirm no domain logic exists in `backend.Application`
- Confirm no DTO classes exist in `backend.Core/Domains/`
- Confirm all controllers inherit from the ABP controller base (not `Controller` directly)

**Expected Result**: Every project contains only the artifact types permitted by the architecture.

**Validation**: No cross-layer violations found. Review passes in under 10 minutes.

**Failure Handling**:

| Symptom | Cause | Corrective Action |
|---|---|---|
| Domain logic found in AppService | Prior dev placed business rules in wrong layer | Extract to a domain service in `backend.Core` |
| DTO found under `backend.Core/Domains/` | Misplaced DTO | Move to `backend.Application/Services/{Entity}Service/DTO/` |
| Controller does not inherit ABP base | Scaffold error | Change base class to `backendControllerBase` |

---

### Step 2 — Replace the EF Core Provider Package

**Description**: Remove `Microsoft.EntityFrameworkCore.SqlServer` from the Infrastructure project
and add `Npgsql.EntityFrameworkCore.PostgreSQL` at the version matching EF Core 9.

**Actions**:
1. Open `backend/src/backend.EntityFrameworkCore/backend.EntityFrameworkCore.csproj`
2. Remove: `<PackageReference Include="Microsoft.EntityFrameworkCore.SqlServer" Version="9.0.5" />`
3. Add: `<PackageReference Include="Npgsql.EntityFrameworkCore.PostgreSQL" Version="9.0.4" />`
4. Remove the three `<Compile Remove="...">` exclusions for deleted SQL Server migration files

**Expected Result**: `.csproj` references Npgsql instead of SqlServer; no stale exclusion lines remain.

**Validation**: `dotnet restore` completes without errors referencing SqlServer packages.

**Failure Handling**:

| Symptom | Cause | Corrective Action |
|---|---|---|
| `package not found` on restore | Version mismatch or NuGet feed issue | Try `9.0.3` or run `dotnet nuget locals all --clear` then retry |
| SqlServer namespace still referenced | Other project also depends on SqlServer | Check `backend.Web.Host.csproj` for any SqlServer reference and remove |

---

### Step 3 — Update `backendDbContextConfigurer.cs`

**Description**: Replace the two `UseSqlServer` calls with `UseNpgsql` so the DbContext registers
the PostgreSQL provider.

**Actions**:
1. Open `backend/src/backend.EntityFrameworkCore/EntityFrameworkCore/backendDbContextConfigurer.cs`
2. Replace `builder.UseSqlServer(connectionString)` with `builder.UseNpgsql(connectionString)`
3. Replace `builder.UseSqlServer(connection)` with `builder.UseNpgsql(connection)`

**Expected Result**: The file uses `UseNpgsql` in both overloads. No other changes needed.

**Validation**: File compiles after the package swap in Step 2.

**Failure Handling**:

| Symptom | Cause | Corrective Action |
|---|---|---|
| `UseNpgsql not found` compile error | Npgsql package not yet restored | Run `dotnet restore` then retry build |
| Namespace missing | `using Npgsql` not auto-resolved | Add `using Microsoft.EntityFrameworkCore;` — `UseNpgsql` is an extension on `DbContextOptionsBuilder` in the Npgsql package |

---

### Step 4 — Delete All Existing Migrations

**Description**: Remove every SQL Server migration file and the model snapshot. These files contain
SQL Server-specific column types and extension method calls that will not compile against Npgsql.

**Actions**:
1. Delete all `.cs` and `.Designer.cs` files under
   `backend/src/backend.EntityFrameworkCore/Migrations/`
2. Delete `backend/src/backend.EntityFrameworkCore/Migrations/backendDbContextModelSnapshot.cs`

**Expected Result**: The `Migrations/` folder is empty.

**Validation**: No `.cs` files remain under `Migrations/`. Solution builds without snapshot errors.

**Failure Handling**:

| Symptom | Cause | Corrective Action |
|---|---|---|
| Build error: `SqlServerModelBuilderExtensions` not found | Snapshot still present | Confirm snapshot file was deleted; check for hidden files |
| EF tooling error: `no snapshot found` | Expected — snapshot was deleted | This is correct; the next step regenerates it |

---

### Step 5 — Update Connection Strings

**Description**: Replace the SQL Server connection strings in both `appsettings.json` files with
the Npgsql PostgreSQL format.

**Actions**:
1. Edit `backend/src/backend.Web.Host/appsettings.json`:
   ```json
   "Default": "Host=localhost;Database=MzansiLegalDb;Username=postgres;Password=<password>"
   ```
2. Edit `backend/src/backend.Migrator/appsettings.json`:
   ```json
   "Default": "Host=localhost;Database=MzansiLegalDb;Username=postgres;Password=<password>"
   ```

**Expected Result**: Both files reference the PostgreSQL host, database name, and credentials.
No SQL Server `Trusted_Connection` or `TrustServerCertificate` attributes remain.

**Validation**: Application loads configuration without throwing `ArgumentException` on the
connection string format.

**Failure Handling**:

| Symptom | Cause | Corrective Action |
|---|---|---|
| `Keyword not supported: 'server'` | Old SQL Server format still present | Replace entire `Default` value string with Npgsql format |
| `28P01: password authentication failed` | Wrong username or password | Verify PostgreSQL user credentials with `psql -U postgres` |
| `3D000: database does not exist` | Database not yet created | The migrator creates the database automatically — this resolves after Step 7 |

---

### Step 6 — Build the Solution

**Description**: Restore NuGet packages and build the full solution to confirm zero compilation
errors after the provider swap.

**Actions**:
```bash
cd backend
dotnet restore
dotnet build
```

**Expected Result**: Build output ends with `Build succeeded. 0 Error(s)`.

**Validation**: Exit code `0`. No `SqlServer*` namespace errors. No missing reference warnings.

**Failure Handling**:

| Symptom | Cause | Corrective Action |
|---|---|---|
| `The type or namespace 'SqlServer' could not be found` | SqlServer package still referenced somewhere | `grep -r "SqlServer" backend/src` to find remaining references |
| `CS0246: type or namespace 'UseNpgsql' not found` | Npgsql package not restored | `dotnet restore --force` |
| Other compilation errors | Unrelated pre-existing issues | Fix each error before proceeding — do not run migrations on a broken build |

---

### Step 7 — Regenerate the Initial Migration

**Description**: Use the EF Core CLI to generate a fresh `InitialCreate` migration targeting
PostgreSQL. This produces correct Npgsql column types and a new model snapshot.

**Actions**:
```bash
cd backend
dotnet ef migrations add InitialCreate \
  --project src/backend.EntityFrameworkCore \
  --startup-project src/backend.Web.Host
```

**Expected Result**:
- `Migrations/` contains `<timestamp>_InitialCreate.cs`, `<timestamp>_InitialCreate.Designer.cs`,
  and `backendDbContextModelSnapshot.cs`
- The migration file uses PostgreSQL column types: `text`, `boolean`, `timestamp without time zone`
- The snapshot uses `NpgsqlModelBuilderExtensions` (not `SqlServerModelBuilderExtensions`)
- `Relational:MaxIdentifierLength` in the snapshot is `63` (PostgreSQL), not `128` (SQL Server)

**Validation**: Open the generated `InitialCreate.cs` and confirm no `nvarchar`, `datetime2`, or
`bit` column types appear.

**Failure Handling**:

| Symptom | Cause | Corrective Action |
|---|---|---|
| `Unable to create a 'DbContext'` | Startup project configuration issue | Ensure `backendDbContextFactory` exists and returns a configured context |
| `No database provider has been configured` | `UseNpgsql` not called in configurer | Re-verify Step 3 changes are saved |
| Migration still shows `nvarchar` / `datetime2` | Wrong startup project used | Explicitly pass `--startup-project src/backend.Web.Host` |
| EF tools not installed | `dotnet ef` command not found | `dotnet tool install --global dotnet-ef` |

---

### Step 8 — Run Migrations Against PostgreSQL

**Description**: Execute the migration runner to apply the `InitialCreate` migration against the
PostgreSQL database, creating the schema and seeding default data.

**Actions**:
```bash
dotnet run --project src/backend.Migrator
```

**Expected Result**:
```
Host database migration started...
Host database migration completed, seeding...
Host database seeds completed.
```

All ABP Zero tables (`AbpUsers`, `AbpRoles`, `AbpSettings`, etc.) created. Default roles
(`Admin`, `Citizen`) and admin user (`admin` / `123qwe`) seeded.

**Validation**: Connect with a PostgreSQL client and run:
```sql
SELECT tablename FROM pg_tables WHERE schemaname = 'public' ORDER BY tablename;
SELECT "Name" FROM "AbpRoles";
SELECT "UserName" FROM "AbpUsers";
```
Confirm ABP tables exist, roles include `Admin` and `Citizen`, and `admin` user is present.

**Failure Handling**:

| Symptom | Cause | Corrective Action |
|---|---|---|
| `Connection refused` | PostgreSQL not running | Start PostgreSQL service |
| `FATAL: role "postgres" does not exist` | Different default user on this system | Update `Username` in connection string to match your PostgreSQL superuser |
| `permission denied to create database` | User lacks CREATE DATABASE rights | Grant with: `ALTER USER postgres CREATEDB;` |
| Migrator exits without seeding | Migration already applied | Expected on re-run — idempotent |

---

### Step 9 — Start the Web Host and Verify Swagger

**Description**: Start the API host and confirm the application is reachable, Swagger UI loads,
and authentication works end-to-end.

**Actions**:
```bash
dotnet run --project src/backend.Web.Host
```

Then verify each endpoint:

1. Open `https://localhost:44311/swagger` — Swagger UI MUST load with ABP endpoints listed
2. Call `GET /api/HealthCheck/Get` via Swagger — MUST return `{ "success": true }`
3. Call `POST /api/TokenAuth/Authenticate` with body:
   ```json
   { "userNameOrEmailAddress": "admin", "password": "123qwe", "rememberClient": false }
   ```
   MUST return a JSON object containing `accessToken`

**Expected Result**: All three checks pass. No runtime exceptions in the console output.

**Validation**:
- Swagger UI loads at `https://localhost:44311/swagger`
- Health check returns HTTP 200 with `{ "success": true }`
- Auth returns HTTP 200 with a non-empty `accessToken`

**Failure Handling**:

| Symptom | Cause | Corrective Action |
|---|---|---|
| `ERR_SSL_PROTOCOL_ERROR` in browser | Dev certificate not trusted | Run `dotnet dev-certs https --trust` |
| `Could not connect to database` at startup | Migrator not yet run | Complete Step 8 first |
| Swagger UI loads but shows no endpoints | ABP module not registered | Verify `backendWebHostModule` depends on `backendWebCoreModule` |
| `401 Unauthorized` on health check | Health check is protected | Use the `accessToken` from auth in Swagger's Authorize dialog |
| `500 Internal Server Error` on authenticate | Seed data missing | Re-run migrator (Step 8) to ensure admin user is seeded |
